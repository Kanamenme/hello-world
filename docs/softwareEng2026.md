本講義を通じて学んだ,ソフトウェア工学の基礎知識からプロジェクトマネジメント,および現代のシステム開発に必須となるバージョン管理とCI/CDの実践について以下にまとめる.

## 1. ソフトウェア工学概論とライフサイクル
ソフトウェアはプログラムだけでなく,データ構造や操作を記述した情報を含むものである.ハードウェアと異なり物理的な劣化はしないものの,社会や技術の変化に追随するための「進化」が求められ,その過程で複雑化するという特徴を持つ.


ソフトウェアライフサイクルは,ビジネスやユーザーのニーズから始まる「計画」「要件定義」「設計」「制作（コーディング）」「テスト」「システムリリース」「保守運用」そして「廃止」という一連の流れで構成される.
特に「要件定義」はプロジェクト関係者間で開発スコープを共有するためのバイブルであり,実現する機能と実現しない機能を明確に切り分ける重要なフェーズである.

## 2. プロジェクトマネジメントとWBS
プロジェクトとは「有期性（必ず終わりがある）」と「独自性（独自の目的を達成する）」を持つ業務である.
プロジェクトを成功に導くためには,現在の立ち位置から順に進む「フォアキャスティング」ではなく,明確なゴールから逆算して必要な要素を計画する「バックキャスティング」の思考が不可欠となる.

この計画を具体化する手法として WBS（Work Breakdown Structure）がある.WBSはプロジェクト目標の達成に必要なすべての作業を、要素成果物を主体に階層的に分解（Breakdown）した構成図である.
作業漏れや重複を防ぐために MECE（漏れなく,ダブりなく）の原則を意識してタスクを洗い出し,スケジュールと担当者（アサイン）を明確に管理する.

## 3. 開発プロセス
代表的な開発プロセスとして,以下の2つのアプローチを学んだ.
* **ウォーターフォールモデル**: 各工程を順番に完了させて後戻りをしない伝統的な手法であり,要件が不変で大規模な開発に適している.
* **アジャイルプロセス**: 変化を歓迎し,数週間単位の短いサイクルで動作するソフトウェアをリリースしていく手法である.「スクラム」「XP（eXtreme Programming）」「カンバン」などの実践手法があり,現代の不確実性の高い開発現場で広く採用されている.

## 4. コーディングと静的解析（PEP8, flake8）
コードは書かれることよりも読まれることの方が多い（Readability counts）ため,読みやすさを担保する共通ルールが必要となる.Pythonにおいては公式のスタイルガイドである「PEP8」がある.
人間が手動ですべてのスタイル違反を取り締まるのは困難であるため,ツールによる自動化（静的解析）が有効である.本演習では flake8を用いて,インデントやコードの長さ,不必要な記号（セミコロン等）などの構文・スタイルエラーを自動検出する手法を学んだ.

## 5. バージョン管理（Git / GitHub）
複数人での共同開発や,安全なコードの変更管理のために Git を用いたバージョン管理手法を修得した.
「main」ブランチから作業用の「dev」などのブランチを切り,コミットを重ねてからリモートリポジトリへプッシュし,「Pull Request（プルリクエスト）」を通じてレビューとマージを行う一連の「GitHubフロー」を実践した.

## 6. CI/CD と GitHub Pages の構築
ソフトウェアを常に動作可能な状態に保つ手法として CI/CD（継続的インテグレーション／継続的デリバリー）を学んだ.

* **CI（継続的インテグレーション）**: コードの変更を検知して自動でビルドやテストを実行する.本演習では GitHub Actions を用い,プルリクエスト作成時に自動で flake8 による構文チェックが走るワークフローを構築した.
* **CD（継続的デリバリー）**: テスト済みのコードを本番環境へ自動デプロイする.本演習では Jekyll を利用し,Markdown ファイルを編集してマージするだけで,GitHub Actions を介して自動的にWebサイト（GitHub Pages）が更新・公開される環境を構築した.
